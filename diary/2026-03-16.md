## 1. 今日の中心テーマ

今日の中心テーマは、

Smolyak 積分器を

「とりあえず動くもの」

から

「高次元・複数精度・長時間実験を支えられる構造」

へ寄せることだった。

問題設定は、単位体積に正規化した立方体

$$
\Omega_d = [-1/2, 1/2]^d
$$

上の積分

$$
I_d(f) = \int_{\Omega_d} f(x)\,dx
$$

である。

この設定にしておくと、

Monte Carlo でも Smolyak でも、

体積係数を別に持たず、

平均や重み付き和がそのまま積分値になる。

この簡潔さは、

実装にも、

実験にも、

理論整理にも効く。

今日は改めて、

この正規化がかなり効いていると感じた。

## 3. 旧版から読めた数値的な傾向

旧版の結果から見えた大枠は明快だった。

### 3.1 `float64`

`float64` はかなり安定していた。

level を上げても、

少なくとも旧版の観測レンジでは、

誤差が自然に下がる方向が見えた。

このことは、

Smolyak そのものが壊れているのではなく、

低精度と加算順が主問題であることを示している。

### 3.2 `float16`, `bfloat16`

`float16` と `bfloat16` は、

高次元・高 level 側でかなり急速に壊れた。

差分則の和は符号が混ざるので、

丸め誤差というより、

相殺誤差が本体である。

感覚的には、

「重み付き和を計算している」

というより、

「正負の近い量の差を何度も取り続けている」

に近い。

このとき、

unit roundoff を $u$ とすると、

単純な逐次加算は概念的に

$$
\hat s_n = (((a_1 + a_2) + a_3) + \cdots + a_n)
$$

の形になり、

誤差は概ね

$$
|\hat s_n - s_n| \lesssim \gamma_{n-1}\sum_{k=1}^n |a_k|,
\qquad
\gamma_m = \frac{mu}{1-mu}
$$

のように見ることができる。

しかも Smolyak では $\sum |a_k|$ が大きく、

真の和 $s_n$ が小さくなることも多い。

この条件では低精度が苦しいのは自然である。

### 3.3 `float32`

`float32` は中間的だった。

初期のレンジではかなり健闘するが、

点数が増えると目に見えて崩れ始める。

ここでも主因は

低精度そのものというより、

低精度で巨大な正負和を処理することである。

## 4. 旧版のボトルネック認識

今日の大きな収穫の一つは、

旧版のボトルネックが

GPU 演算ではなく、

host 側の格子構築と保持だと確信できたことだ。

もともと感触としてはそうだったが、

JSONL とログと HLO 観察が揃ったことで、

かなり確信に近づいた。

旧版では、

- term ごとに tensor product を展開し、
- 点列をまとめ、
- global な `unique` 相当で統合し、
- points と weights を明示的に保持する、

という流れが強かった。

この構造だとメモリ量は直観的に

$$
M \approx N_{\mathrm{pts}} \, d \, b_x
  + N_{\mathrm{pts}} \, b_w
  + M_{\mathrm{temp}}
$$

となる。

ここで

- $N_{\mathrm{pts}}$ は保持点数
- $d$ は次元
- $b_x$ は座標の 1 要素あたりバイト数
- $b_w$ は重みの 1 要素あたりバイト数
- $M_{\mathrm{temp}}$ は `meshgrid`, `unique`, index 配列などの一時領域

である。

特に問題なのは $M_{\mathrm{temp}}$ で、

本体より一時領域の方が重くなる。

今日は改めて、

本質的な問題は

「GPU でどれだけ速く dot するか」

ではなく、

「GPU に渡す前の host 処理をどう薄くするか」

だと確認できた。

## 5. HLO の意味づけ

HLO 観察も進めた。

ここで重要だったのは、

モジュール内で JIT を入れ込むのではなく、

実験コード側で `integrate(f, integrator)` を包んで JIT し、

Python 再帰をアンローリングさせる方針にしたことだ。

この方針はかなり良かった。

理由は二つある。

一つは可読性。

再帰は積分の構造をかなり素直に表せる。

もう一つは実験の自由度。

JIT 境界を実験コード側に置くと、

何を lower するかを実験ごとに変えられる。

HLO を見た結果、

支配的なのは大きい `dot` よりも、

- `while`
- `gather`
- `slice`
- `reshape`
- `broadcast_in_dim`

だった。

つまり、

線形代数の重い本体が GPU を占有しているというより、

index 処理とデータ組み立てが前面に出ている。

この読みは、

GPU 使用率が低いこととも整合的である。

## 6. なぜ再帰を残すのか

今日の中で一番何度も考えたのは、

再帰をどこまで残すかだった。

結論としては、

再帰は残してよい、

ただし JIT は実験コード側、

でいくのが今は最善だと考えている。

Smolyak の構造を考えると、

外側では term を回し、

内側では固定次元の tensor product をたどる。

ここで次元 $d$ が固定なら、

Python 再帰は実質

「静的な展開記法」

として使える。

もちろん、

HLO サイズは増える。

しかし今のところ、

サイズそのものより、

実装の見通しと host メモリ削減の方が重要である。

今日はユーザーともその認識を共有できたのがよかった。

## 7. 新しい設計イメージ

新しい設計の核は、

explicit mesh を作らないことだ。

持つべきものは、

巨大な `points, weights` ではなく、

- level ごとの 1D rule
- Smolyak term の multi-index
- 必要なら term coefficient

である。

記号で書くと、

Smolyak 型の近似は

$$
A(q,d)
=
\sum_{|i|_1 \le q + d - 1}
(\Delta_{i_1} \otimes \cdots \otimes \Delta_{i_d})
$$

あるいは combination technique 的には

$$
A(q,d)
=
\sum_{|i|_1 \in \mathcal{I}(q,d)}
c_i

(Q_{i_1} \otimes \cdots \otimes Q_{i_d})
$$

のように書ける。

いま欲しいのは、

この和を

「全点を先に materialize してから一度に計算する」

のではなく、

「term ごと・prefix ごとに評価して足し込む」

実装である。

この方向へ進むと、

メモリ保持量は大きく減る。

## 8. 末端 1 次元を batched 評価する構想

今日の議論の中で特に自然だったのは、

固定次元の再帰を辿り、

最後の 1 次元だけを batched 評価する案だった。

たとえば、

prefix を $x_{1:d-1}$ と固定すれば、

末端では

$$
\sum_{j=1}^{n_d}
w^{(d)}_j
f(x_1,\dots,x_{d-1}, t_j)
$$

を計算すればよい。

この部分は

- 点列生成
- `vmap(f)`
- `dot` or `tensordot`

に自然に落ちる。

つまり、

「上位は木構造」

「末端は線形代数」

というハイブリッドになる。

この構造はかなり筋が良い。

完全な逐次 1 点評価より GPU に優しいし、

全点展開よりメモリに優しい。

## 9. 1D rule をどう持つか

もう一つの重要論点は、

1D rule をどう持つかだった。

議論の中で、

「dict 的に level ごとに rule を持つ」

という発想が出た。

これは API 的にはとても自然だと思う。

最大 level を先に決めておけば、

$$
\{1 \mapsto Q_1,\ 2 \mapsto Q_2,\ \dots,\ L_{\max}\mapsto Q_{L_{\max}}\}
$$

という対応を最初に準備しておける。

精度を上げたいときは、

active level だけ上げればよい。

このとき、

prepared level $L_{\max}$ と

active level $\ell$ を分ける考え方はかなりよい。

$$
1 \le \ell \le L_{\max}
$$

であれば、

再初期化なしに refinement できる。

今日はこの設計にかなり納得感が出た。

## 10. ID と重複判定の理解

ノード識別についても整理が進んだ。

重要なのは、

「全空間で一意な通し番号」

を最初から必須にしないことである。

まず rule 参照には、

`(level, index_in_level)`

で十分である。

一方で、

nested rule 間の幾何学的な重複判定には、

canonical な node ID が必要になる。

この二段階の理解は大事だった。

つまり、

- 参照のための ID
- 同一性判定のための ID

は分けてよい。

最初からすべてを厳密な global canonical ID で回そうとすると、

かえって実装が重くなる。

今日はそこをかなりはっきり整理できた。

## 11. `uint32` の感触

ID 表現については、

`uint32` がかなり現実的だという感触も持てた。

少なくとも rule 内の local index であれば、

極端な level を理論上だけ考えなければ、

`uint32` は相当広いレンジを持てる。

ただし、

offset や総点数まで軽率に `uint32` に落とすのは危ない。

ここは

- local node index: 軽い整数
- 総点数や offset: 安全側

という分離が自然だと確認した。

こういう細部の整理は、

あとで実装の迷いを減らす。

## 15. 数式をメモに書けるようにした意味

今日ユーザーから、

`notes` と `diary` にも数式を書けるようにしたい、

という話があった。

これは実際かなり重要だと思う。

Smolyak の議論は、

文章だけだとどうしても曖昧になる。

たとえば、

誤差要因の分解一つ取っても、

$$
e_{\mathrm{total}}
=
e_{\mathrm{quadrature}}
+
e_{\mathrm{rounding}}
+
e_{\mathrm{accumulation}}
+
e_{\mathrm{implementation}}
$$

のような構図を書けるだけで、

考え方がかなり揃う。

同じように、

解析解を

$$
\int_{\Omega_d} \exp(a^\top x)\,dx
=
\prod_{k=1}^d \frac{2\sinh(a_k/2)}{a_k}
$$

と明示できると、

何を比較しているかが一気に分かりやすくなる。

今後、

数値線形代数や最適化の話でも、

これは効いてくるはずだ。
