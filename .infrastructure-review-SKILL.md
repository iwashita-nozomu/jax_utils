# Infrastructure Review スキル

**目的**: プロジェクトの Docker 環境・依存関係・CI/CD パイプラインの**品質と一貫性**を担保する。

**【スコープ】インフラストラクチャ設定・環境構築のレビューのみ**（アプリケーション・コードレビューは対象外）

**対象ファイル**:
- Docker: `Dockerfile`, `docker-compose.yml`, `docker/**`
- Python 依存関係: `docker/requirements.txt`, `requirements*.txt`
- CI/CD: `.github/workflows/*.yml`, `scripts/docker*.sh` 等

**レビュー深度**: 詳細レビュー
- Docker ベストプラクティス（キャッシュ戦略、多段ビルド）
- 依存関係（バージョンピニング、互換性）
- セキュリティ（不要な権限、secrets 管理）
- パフォーマンス（イメージサイズ、ビルド時間）
- ドキュメンテーション（セットアップ手順の明確性）

## 1. Dockerfile レビュー

### 1.1 基本チェック項目

#### ベースイメージ
- [ ] ベースイメージが明示的に指定されているか（`latest` tag 不可）
- [ ] Python バージョンが明確か（例: `python:3.9` ではなく `python:3.9.18`）
- [ ] ベースイメージが official image か（Alpine/Debian を選択できているか）
- [ ] multi-stage build が活用されているか（本番イメージサイズを最小化）

**チェックコード例**:
```dockerfile
# ❌ NG: latest tag、バージョン不明確
FROM python:latest
RUN pip install -r requirements.txt

# ✅ OK: 明確なバージョン、multi-stage ビルド
FROM python:3.9.18-slim as builder
WORKDIR /build
COPY docker/requirements.txt .
RUN pip install --user -r requirements.txt

FROM python:3.9.18-slim
COPY --from=builder /root/.local /root/.local
ENV PATH=/root/.local/bin:$PATH
```

#### ワーキングディレクトリ
- [ ] `WORKDIR` が明示的に設定されているか
- [ ] パスが絶対パスか（相対パスは避ける）
- [ ] または複数回の `WORKDIR` が論理的に整理されているか

#### ユーザー権限
- [ ] root で実行していないか（セキュリティリスク）
- [ ] `USER` ディレクティブで非 root ユーザーに切り替えているか

**チェックコード例**:
```dockerfile
# ❌ NG: root で実行
RUN adduser --home /home/app app
WORKDIR /home/app
# （ユーザー切り替えなし）

# ✅ OK: 非 root ユーザーに切り替え
RUN groupadd -r app && useradd -r -g app app
WORKDIR /app
COPY --chown=app:app . .
USER app
```

#### キャッシュ戦略
- [ ] 変更頻度の低いステップが前に配置されているか
- [ ] `pip install` は、`requirements.txt` が最後に更新された場合のみ再実行されるか
- [ ] 不要なファイル（テスト、ドキュメント）が `COPY` されていないか（`.dockerignore` で除外）

**チェックコード例**:
```dockerfile
# ❌ NG: コード変更時に pip reinstall（遅い）
COPY . .
RUN pip install -r docker/requirements.txt

# ✅ OK: requirements.txt のみ先に COPY
COPY docker/requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

#### レイヤー最小化
- [ ] `RUN` コマンドが過剰に分割されていないか（`&&` で連結）
- [ ] 不要な中間ファイルが削除されているか（例: `apt-get clean`）

**チェックコード例**:
```dockerfile
# ❌ NG: 複数レイヤーに分割（イメージサイズ増加）
RUN apt-get update
RUN apt-get install -y build-essential
RUN rm -rf /var/lib/apt/lists/*

# ✅ OK: 1 レイヤーで実行、クリーンアップ
RUN apt-get update && \
    apt-get install -y build-essential && \
    rm -rf /var/lib/apt/lists/*
```

### 1.2 健全性チェック

#### エントリーポイント / コマンド
- [ ] `ENTRYPOINT` が設定されているか（スクリプトまたはコマンド）
- [ ] Docker コンテナが正常に終了するか（ゾンビプロセス対策）
- [ ] 引数の受け渡しが正確か（`ENTRYPOINT` + `CMD` の組み合わせ）

#### ホルムアップ・ヘルスチェック
- [ ] アプリケーションの起動時間が合理的か
- [ ] `HEALTHCHECK` が定義されているか（長時間実行プロセスの場合）

**チェックコード例**:
```dockerfile
# ❌ NG: ヘルスチェックなし、終了処理不明確
CMD python app.py

# ✅ OK: ヘルスチェック + exec form
ENTRYPOINT ["python", "-u", "app.py"]
HEALTHCHECK --interval=30s --timeout=10s \
  CMD python -c "import requests; requests.get('http://localhost:8000/health')"
```

## 2. python 依存関係（requirements.txt）レビュー

### 2.1 形式・完全性

#### 明示的なバージョン指定
- [ ] すべてのパッケージにバージョンが指定されているか（`==` を使用）
- [ ] 互換性のあるバージョン（`~=`, `>=` 等）は例外的か
- [ ] 依存関係の推移的な依存関係が文書化されているか

**チェックコード例**:
```txt
# ❌ NG: バージョンなし、互換性記号
jax
jaxlib>=0.3.0
numpy

# ✅ OK: 明示的なバージョン + コメント
jax==0.4.20
jaxlib==0.4.20
numpy==1.24.3  # データ型互換性のため、numpy 2.0+ は非対応
pytest==7.4.3
```

#### コメントと説明
- [ ] 各パッケージに簡潔なコメントがあるか（特に非標準パッケージ）
- [ ] バージョン選択の理由が記載されているか（互換性、セキュリティ等）
- [ ] オプション依存関係が別セクションに分けられているか

**チェックコード例**:
```txt
# Mandatory dependencies
jax==0.4.20
jaxlib==0.4.20

# Optional: for experiments
# To use: pip install -r requirements.txt -e ".[experiments]"
# [experiments]
# smolyak==2.1.0
```

### 2.2 セキュリティ・最適化

#### セキュリティアップデート
- [ ] 既知の脆弱性( CVE)のあるバージョンが避けられているか
- [ ] 定期的に更新されているか（最終更新日の記録）

#### マルチプラットフォーム対応
- [ ] OS / Python バージョン別の条件付けがあるか（必要な場合）
- [ ] プリコンパイル済みホイール(wheel)が利用可能か

**チェックコード例**:
```txt
# Python 3.9+ required
# Platform-specific dependencies
numpy==1.24.3; python_version>="3.9"
scipy==1.11.0

# If Windows
; platform_system == 'Windows'
pywin32==305
```

## 3. docker-compose.yml レビュー

### 3.1 基本チェック

#### サービス定義
- [ ] すべてのサービスに `image` または `build` が指定されているか
- [ ] ネットワーク設定が明確か（デフォルト bridge を使用）
- [ ] ボリュームマウント（volumes）が適切か

#### 環境変数
- [ ] `.env` ファイルで環境変数を外部化しているか（credentials 禁止）
- [ ] `environment` セクションでデフォルト値が設定されているか

**チェックコード例**:
```yaml
# ❌ NG: credentials がハードコード、environment がない
version: '3'
services:
  app:
    image: my-app:latest
    environment:
      - DB_PASSWORD=secret123

# ✅ OK: .env で管理、デフォルト値設定
version: '3.9'
services:
  app:
    build: .
    environment:
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
      - DB_HOST=${DB_HOST:-postgres}
    env_file: .env
```

## 4. CI/CD パイプライン（GitHub Actions）レビュー

### 4.1 ワークフロー定義

#### トリガーイベント
- [ ] `on:` セクションで適切なトリガーが設定されているか（push, pull_request 等）
- [ ] ブランチ条件（`branches:` 等）が設定されているか

#### ジョブ定義
- [ ] `runs-on:` が明確か（`ubuntu-latest` の固定化）
- [ ] 複数 OS / Python バージョンでの matrix テストが用意されているか

**チェックコード例**:
```yaml
# ❌ NG: トリガー不明確、Python バージョン固定
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
      - run: python --version  # 何もテストしていない

# ✅ OK: 複数バージョンでの matrix テスト
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
        python-version: ["3.9", "3.10", "3.11"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r docker/requirements.txt
      - run: pytest
```

#### Secrets 管理
- [ ] 機密情報（API キー、credentials）がハードコードされていないか
- [ ] `secrets.*` を使用しているか

## 5. Docker 環境の整合性チェック

### 5.1 Dockerfile ⟷ requirements.txt ⟷ CI/CD の同期

#### 依存関係の一貫性
- [ ] `docker/requirements.txt` の Python パッケージが、実装で実際に使用されているか
- [ ] CI/CD の `pip install` コマンドが、`docker/requirements.txt` と同じバージョンを使用しているか

**チェック例**:
```bash
# grep で使用パッケージを確認
grep "^import\|^from" python/**/*.py | sed 's/.*import //' | sort -u

# requirements.txt と比較
diff <(grep -E "^[a-zA-Z]" docker/requirements.txt | cut -d= -f1 | sort) \
     <(上記 grep 結果)
```

#### PYTHONPATH の明示
- [ ] Dockerfile で `ENV PYTHONPATH=/workspace/python` が設定されているか
- [ ] CI/CD で同じ `PYTHONPATH` が使用されているか

### 5.2 イメージビルド・テスト

#### ビルド検証
- [ ] `docker build` が成功するか（構文エラーなし）
- [ ] イメージサイズが合理的か（compressed size < 1GB for Python apps）
- [ ] ビルド時間が過度でないか

**実行例**:
```bash
# Dockerfile 構文チェック
hadolint Dockerfile

# イメージサイズ確認
docker build -t test-image . && docker images test-image --format "table {{.Repository}}\t{{.Size}}"
```

#### コンテナ起動テスト
- [ ] コンテナが正常に起動するか
- [ ] main command が正確に実行されるか
- [ ] ヘルスチェック（HEALTHCHECK）が通るか

**実行例**:
```bash
docker run --rm test-image python --version
docker run --rm test-image python -c "import jax; print(jax.__version__)"
```

## 6. セキュリティ・ベストプラクティス

### 6.1 イメージセキュリティ

#### ベースイメージの選択
- [ ] 最小限の`slim` または `alpine` イメージが使用されているか
- [ ] セキュリティアップデートが頻繁なディストリビューション（Ubuntu, Debian）いるか
- [ ] 非 root ユーザーで実行されているか

#### 脆弱性スキャン
- [ ] `trivy` または `grype` でコンテナイメージをスキャンしているか
- [ ] 既知の CVE がないか

**実行例**:
```bash
# Trivy でイメージスキャン
trivy image my-app:latest

# Grype でスキャン
grype my-app:latest -o table
```

## 7. チェックリスト（実装段階別、優先度付き）

### 🔴 Critical（必須）

- [ ] Dockerfile が存在し、構文エラーがない
- [ ] `python/requirements.txt` または `docker/requirements.txt` が存在
- [ ] すべての Python パッケージにバージョンが指定（`==`）
- [ ] Docker イメージの non-root ユーザー化
- [ ] PYTHONPATH が Dockerfile で明示

### 🟠 Major（推奨）

- [ ] Multi-stage build が活用されている
- [ ] `.dockerignore` でテスト・ドキュメントが除外
- [ ] キャッシュ戦略が最適化（requirements.txt が先に COPY）
- [ ] `HEALTHCHECK` が定義（長時間実行プロセス）
- [ ] GitHub Actions で複数 Python バージョンテスト

### 🟡 Minor（次回要可）

- [ ] Hadolint で Dockerfile lint チェック
- [ ] イメージサイズが 1GB 未満
- [ ] CVE スキャン（trivy / grype）
- [ ] README に Docker セットアップ手順が記載
- [ ] docker-compose.yml で `.env` 使用

## 9. Infrastructure Review 実施フロー

### PR 作成前（開発者）

1. Docker ビルド・テスト実行確認
2. `pip install -r docker/requirements.txt` で新パッケージが正常にインストール
3. `docker run my-app python -c "import module"` で動作確認

### レビュー時（レビュアー）

1. Dockerfile 構文チェック（`hadolint`）
2. requirements.txt バージョン指定確認
3. doctor-compose.yml での environment / secrets 使用確認
4. GitHub Actions ワークフロー確認

### Merge 前（Maintainer）

1. イメージビルド検証
2. CVE スキャン実行（`trivy`）
3. ヘルスチェック・起動テスト
